# Codebase Concerns

**Analysis Date:** 2026-03-24

## Critical Issues

### Tax Rate Data Inconsistencies Between `states.json` and `taxRules.ts`

The hardcoded tax rules in `src/lib/taxRules.ts` contradict the data extracted into `src/data/states.json` (generated by `scripts/build-data.mjs`) for several states. These are the two independent sources of truth that should agree but do not.

**Alabama (AL) grocery rate:**
- Issue: `states.json` groceryNote says "reduced state rate of 2%", but `src/lib/taxRules.ts` line 66 has `AL: { rate: 0.03 }` (3%)
- Files: `src/lib/taxRules.ts` (line 66), `state-taxes/AL.md`
- Impact: Calculator charges 3% on groceries instead of the correct 2% rate
- Fix approach: Update `GROCERY_REDUCED_RATE.AL.rate` to `0.02` in `src/lib/taxRules.ts`

**Arkansas (AR) grocery rate:**
- Issue: `states.json` groceryNote says "Exempt from state sales tax as of January 1, 2026", but `src/lib/taxRules.ts` line 67 still has `AR: { rate: 0.0065 }` (0.125% state rate). Since AR groceries became exempt in Jan 2026, the calculator is applying an outdated reduced rate.
- Files: `src/lib/taxRules.ts` (line 67), `state-taxes/AR.md`
- Impact: Calculator charges 0.125% on AR groceries that should now be exempt
- Fix approach: Remove AR from `GROCERY_REDUCED_RATE` and ensure grocery exemption logic handles it correctly

**Kansas (KS) grocery rate:**
- Issue: `states.json` groceryNote says "State sales tax rate on food is 0% as of January 1, 2025", but `src/lib/taxRules.ts` line 71 has `KS: { rate: 0.04 }` (4%). The KS grocery tax phase-out reached 0% in 2025.
- Files: `src/lib/taxRules.ts` (line 71), `state-taxes/KS.md`
- Impact: Calculator charges 4% on KS groceries that should be exempt
- Fix approach: Remove KS from `GROCERY_REDUCED_RATE`; groceries should fall through to the exempt path

**Mississippi (MS) grocery rate:**
- Issue: `states.json` groceryNote says "Taxed at a reduced rate of 5%", but `src/lib/taxRules.ts` line 72 has `MS: { rate: 0.07 }` (7%, the full state rate). Additionally, MS appears in both `GROCERY_REDUCED_RATE` and `GROCERY_FULL_TAX` sets, creating ambiguous logic.
- Files: `src/lib/taxRules.ts` (lines 72, 80), `state-taxes/MS.md`
- Impact: Calculator charges full 7% instead of the correct reduced 5%. The `GROCERY_FULL_TAX` set takes precedence in the switch statement (line 150), so the `GROCERY_REDUCED_RATE` entry for MS is dead code.
- Fix approach: Remove MS from `GROCERY_FULL_TAX`; update `GROCERY_REDUCED_RATE.MS.rate` to `0.05`

**Idaho (ID) grocery status:**
- Issue: `states.json` says `groceries: "exempt"` (parsed from markdown), but `src/lib/taxRules.ts` has ID in `GROCERY_FULL_TAX` (line 80) and in `GROCERY_REDUCED_RATE` with rate `0.06` (line 69). Idaho taxes groceries at the full rate but provides an income tax credit. The `states.json` classification as "exempt" is incorrect.
- Files: `src/lib/taxRules.ts` (lines 69, 80), `scripts/build-data.mjs` (exemption parsing), `state-taxes/ID.md`
- Impact: The state detail page (`StatePage.tsx`) shows "Exempt" badge for ID groceries, while the calculator correctly taxes them at 6%. User sees contradictory information.
- Fix approach: Fix the markdown parsing in `build-data.mjs` or correct the ID markdown source to use "taxable" status

## Technical Debt

### Duplicated `StateEntry` Interface Definition — Medium
- Issue: The TypeScript interface for state data is defined independently in three places with different subsets of fields: `src/pages/HomePage.tsx` (lines 5-16), `src/pages/StatePage.tsx` (lines 5-37), and `src/lib/taxRules.ts` (lines 11-29 as `StateData`). No shared type definition exists.
- Files: `src/pages/HomePage.tsx`, `src/pages/StatePage.tsx`, `src/lib/taxRules.ts`
- Impact: Adding a new field to states.json requires updating multiple files. Divergent interfaces can silently miss fields.
- Fix approach: Create a single `StateEntry` type in `src/data/types.ts` and import it everywhere

### Dead Code: `useTheme` Hook — Low
- Issue: `src/lib/useTheme.ts` exports a `useTheme()` hook, but it is never imported anywhere. `src/components/Layout.tsx` implements identical dark mode logic inline (lines 5-16).
- Files: `src/lib/useTheme.ts`, `src/components/Layout.tsx`
- Impact: Minor clutter. The two implementations could drift.
- Fix approach: Either delete `useTheme.ts` or refactor `Layout.tsx` to use it

### Hardcoded Tax Rules Separate from Data Pipeline — Medium
- Issue: Tax exemption logic in `src/lib/taxRules.ts` uses hardcoded sets (`CLOTHING_FULLY_EXEMPT`, `CLOTHING_PARTIAL`, `GROCERY_REDUCED_RATE`, `GROCERY_FULL_TAX`, `CANDY_TAXABLE_WITH_FOOD_EXEMPT`) that are maintained independently from the markdown source files and the `build-data.mjs` pipeline. The data pipeline extracts exemption statuses, but the calculator ignores them and uses its own constants.
- Files: `src/lib/taxRules.ts`, `scripts/build-data.mjs`, `state-taxes/*.md`
- Impact: Two sources of truth that easily go out of sync (as demonstrated by the critical issues above). When tax laws change, both the markdown files AND `taxRules.ts` must be updated independently.
- Fix approach: Either derive the calculator rules from `states.json` data at build time, or consolidate all rate data into the markdown files and have `build-data.mjs` extract specific rates for each category

### Full Markdown Embedded in JSON Bundle — Medium
- Issue: `states.json` includes a `markdownContent` field for every state containing the full source markdown. This adds ~273KB of raw text (82% of the 348KB file) to the client bundle. The `markdownContent` field is never referenced in any client-side code.
- Files: `scripts/build-data.mjs` (line 742), `src/data/states.json`
- Impact: The built JS bundle is 590KB, of which a large portion is unused markdown text shipped to every visitor
- Fix approach: Remove `markdownContent` from the JSON output in `build-data.mjs`, or lazy-load markdown content only on the state detail page

## Performance Concerns

**Large initial bundle size:**
- Problem: The single JS chunk is 590KB (uncompressed) primarily because `states.json` (348KB with embedded markdown) is imported synchronously in three page components
- Files: `dist/assets/index-DghLmqbi.js` (604KB), `src/data/states.json` (348KB)
- Cause: No code splitting, no lazy loading of routes, all state data loaded eagerly including unused `markdownContent` field
- Improvement path: (1) Remove `markdownContent` from JSON; (2) Add React.lazy() for route-level code splitting; (3) Consider loading state detail data on demand

**No route-level code splitting:**
- Problem: All three pages (HomePage, StatePage, CalculatorPage) are bundled into a single chunk
- Files: `src/App.tsx`
- Fix: Use `React.lazy()` + `Suspense` for `StatePage` and `CalculatorPage`

## Scalability

**Static data only:**
- Limitation: All tax data is compiled at build time from markdown files. There is no mechanism to update rates without a full rebuild and deploy.
- Impact if growth occurs: If the site needs to cover more jurisdictions (e.g., local taxes for ~11,000 US tax jurisdictions), the current architecture of one JSON blob loaded eagerly will not scale.

## Missing Infrastructure

**No tests:**
- What's missing: Zero test files exist in the project. No test framework is configured (no jest/vitest config).
- Why it matters: The tax calculation engine in `src/lib/taxRules.ts` contains complex conditional logic for 50+ jurisdictions with category-specific exemptions, threshold-based partial exemptions, and multiple rate tiers. This is exactly the kind of logic that needs comprehensive unit tests. The data inconsistencies listed above would be caught by tests comparing expected tax amounts against known-correct values.
- Priority: High. Add vitest and write tests for `calculateTax()` covering each state and category combination.

**No linting or formatting configuration:**
- What's missing: No ESLint, Prettier, or Biome config files exist
- Why it matters: No automated style enforcement; code consistency depends entirely on manual discipline

**No 404.html for GitHub Pages SPA routing:**
- What's missing: The `dist/` directory has no `404.html`, which GitHub Pages needs to serve the SPA for deep links (e.g., `/us-tax/state/CA`). Users who directly navigate to or refresh a non-root URL will get a GitHub Pages 404 error.
- Files: `vite.config.ts`, `.github/workflows/deploy.yml`
- Fix approach: Add a post-build step that copies `dist/index.html` to `dist/404.html`

**No catch-all route for unknown paths:**
- What's missing: `src/App.tsx` defines only three routes. Any other path renders an empty `<Layout>` with no content and no indication that the page was not found.
- Files: `src/App.tsx`
- Fix approach: Add a `<Route path="*" element={<NotFoundPage />} />` inside the layout route

**No error boundary:**
- What's missing: No React error boundary wraps the app. If a rendering error occurs (e.g., the non-null assertion on line 59 of `CalculatorPage.tsx` fails), the entire app crashes to a white screen.
- Files: `src/App.tsx`, `src/main.tsx`

**No `<meta>` description or Open Graph tags:**
- What's missing: `index.html` has only a `<title>` tag. No meta description, no OG tags for social sharing.
- Files: `index.html`

## Code Smells

**Non-null assertion on potentially undefined lookup:**
- Where: `src/pages/CalculatorPage.tsx` line 59: `const state = states.find((s) => s.abbreviation === selectedState)!;`
- Issue: If `selectedState` is set to an invalid value (e.g., via URL manipulation of the `?state=` param), `.find()` returns `undefined` and the `!` assertion silently allows it through. The subsequent `useMemo` on line 61 checks `if (!state)` but the `!` assertion undermines TypeScript's ability to catch this.
- Suggested improvement: Remove the `!` and add proper null handling, or validate `initialState` against the states list

**Unreachable code in candy case:**
- Where: `src/lib/taxRules.ts` lines 199-203
- Issue: The `if (!state.hasSalesTax)` check inside the candy `case` can never be true because line 96 already returns early if `!state.hasSalesTax`
- Suggested improvement: Remove the dead branch

**`states.json` is gitignored but required for build:**
- Where: `.gitignore` line 3: `src/data/states.json`
- Issue: The generated JSON file is in `.gitignore` (correctly, since it is generated), but no documentation or `README` explains that `npm run build:data` must be run before development. The `dev` and `build` scripts do run `build-data.mjs` first, so this works in practice, but IDE tooling may show import errors until the first build.
- Suggested improvement: Add a note in README or add a `postinstall` script

## Data Accuracy

**Multiple grocery tax rates are stale or wrong (see Critical Issues above):**
- AL: Calculator uses 3% but correct rate is 2% (as of Sep 2025)
- AR: Calculator uses 0.125% reduced rate but groceries became exempt Jan 2026
- KS: Calculator uses 4% reduced rate but groceries became exempt Jan 2025
- MS: Calculator uses 7% (full rate) but correct reduced rate is 5%
- ID: State info page says "exempt" but calculator correctly taxes at 6%

**No automated verification of tax data:**
- There is no mechanism to validate that the hardcoded rates in `taxRules.ts` match current law. Tax rates change frequently (e.g., KS and AR both had grocery tax changes take effect recently). Without tests or a single source of truth, stale data will accumulate.

**No "last updated" or "effective date" indicator:**
- The site shows tax information but does not indicate when the data was last verified or what effective date the rates apply to. The footer disclaimer says "Tax rates and rules change frequently" but provides no date context.

**Floating-point arithmetic in tax calculations:**
- `src/lib/taxRules.ts` computes `taxableAmount * taxRate` using native JavaScript floating-point multiplication (e.g., `35.00 * 0.01 = 0.35000000000000003`). While `Intl.NumberFormat` in `formatCurrency()` rounds the display output correctly, the intermediate `taxAmount` values stored in `TaxLineItem` objects carry floating-point noise. Summing multiple noisy line items could produce a total that rounds differently than summing the individually rounded amounts.
- Files: `src/lib/taxRules.ts` (line 235), `src/lib/formatters.ts`
- Impact: For this demo calculator, the visual impact is negligible (Intl rounds display). But for any real tax compliance use, amounts should be rounded to cents at each line item before summing.
- Fix approach: Round `taxAmount` to 2 decimal places on line 235: `Math.round(taxableAmount * taxRate * 100) / 100`

## Test Coverage Gaps

- What's not tested: The entire application has zero tests. Most critically, `src/lib/taxRules.ts` (the `calculateTax` function) handles complex multi-state, multi-category tax logic with threshold-based partial exemptions and should have comprehensive unit tests.
- Files: `src/lib/taxRules.ts`, `src/lib/formatters.ts`, `scripts/build-data.mjs`
- Risk: Tax calculation errors go undetected. The critical data inconsistencies listed above demonstrate this risk.
- Priority: High
